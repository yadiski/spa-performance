# CSP Hardening Runbook

## Current status

The API ships a strict Content Security Policy (CSP) via the `securityHeaders`
middleware (`apps/api/src/http/security-headers.ts`).

| Environment | Header name | Mode |
|---|---|---|
| `production` | `Content-Security-Policy` | **Enforcing** |
| `development` / `test` | `Content-Security-Policy-Report-Only` | Report-only |

## Policy directives

```
default-src 'self';
script-src 'self';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self' https://api.openrouter.ai https://api.resend.com;
frame-ancestors 'none';
base-uri 'self';
form-action 'self';
```

No `'unsafe-inline'` is used for script-src or style-src.

## Why no `'unsafe-inline'`

Vite's production build emits all scripts and styles as external hashed files
(JS chunks + CSS files). Inline scripts are not generated by default. The project's
`vite.config.ts` does not enable `build.inlineDynamicImports`, so this assumption
holds.

If a future Vite plugin or SSR setup introduces inline scripts, the correct fix is:
1. Extract the inline content to an external file, OR
2. Hash the known-safe inline block (sha256, base64-encoded) and pass it to
   `cspScriptHashes` / `cspStyleHashes` in the `securityHeaders()` call in
   `apps/api/src/http/app.ts`.

## Ramp-up procedure for new violations

1. Temporarily switch to report-only mode in production:
   - In `apps/api/src/http/app.ts`, pass `{ cspReportOnly: true }` to `securityHeaders()`.
   - Deploy.
2. Add a CSP violation report endpoint or use an external service (e.g. report-uri.com).
   - Add `report-uri /api/v1/csp-report` (or an external URI) to the CSP string in
     `buildCsp()` inside `security-headers.ts`.
3. Collect violations for 24–72 hours in production traffic.
4. For each legitimate violation:
   - If it's a first-party resource: move it to an external file.
   - If it must be inline (rare): compute `sha256` of the exact content, base64-encode
     it, add `'sha256-<base64>'` to `cspScriptHashes` or `cspStyleHashes`.
5. Switch back to enforcing mode and re-deploy.

## Adding inline script hashes

Compute the hash of the exact script content (including whitespace):

```sh
echo -n '<script content here>' | openssl dgst -sha256 -binary | base64
```

Then pass the token in `app.ts`:

```ts
app.use('*', securityHeaders({
  cspScriptHashes: ["sha256-<base64_output>"],
}));
```

## Additional connect-src origins

To allow the front-end to connect to additional external APIs, pass them to
`connectSrcExtra`:

```ts
app.use('*', securityHeaders({
  connectSrcExtra: ['https://analytics.example.com'],
}));
```
